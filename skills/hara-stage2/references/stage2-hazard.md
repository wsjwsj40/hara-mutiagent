# Stage 2：整车危害映射方法

本文件只定义功能故障到整车危害的语义映射方法，不定义 JSON schema。输出结构以 `json-contracts.md` 为准。

## 目标

将当前 Stage1 单功能片段中非 `nan` 且 `field_reasoning.推理.是否有安全风险=是` 的功能故障，映射为车辆层面的危害。

## 基本规则

- 展开字段：`功能丧失`、`过大`、`过早`、`过小`、`过晚`、`非预期激活`、`卡滞`、`方向错误`。
- 一个非 `nan` 且 `是否有安全风险=是` 的故障单元格对应一条 MF。
- 非 `nan` 但 `是否有安全风险=否` 的故障不生成 Stage2 行；它们只作为 Stage1 功能故障留痕。
- `整车级危害` 必须逐字来自 `vehicle_hazards.json`。
- `故障描述` 来自 Stage1；本文件只定义整车危害映射，不定义故障描述生成规则。
- 每条 MF 必须保留 Stage1 追溯信息，便于 Stage3 回到 Stage0 `detail_text`。

## 映射步骤

1. 识别功能本意：该功能原本要让车辆或系统实现什么？
2. 识别功能影响：故障后输出、状态、时机或方向发生什么异常？
3. 推导车辆级后果：车辆运动或状态实际会发生什么变化？
4. 区分关键对立面：例如“自己动了”还是“想动动不了”，“意外减速”还是“减速能力丧失”。
5. 从 `vehicle_hazards.json` 中选择最贴近的危害。
6. 在 `hazard_reasoning` 中记录功能影响、车辆级后果、关键判断和选择理由。

## 核心判断逻辑

### 功能本意决定判断方向

选择整车危害时，必须从”功能本意”出发，而非从”故障现象”出发：

1. **该功能原本要让车辆做什么？**
   - 让车辆动（驱动）？还是让车辆不动（保持/约束）？
   - 让车辆加速？还是让车辆减速？
   - 让车辆保持某状态？还是改变车辆状态？

2. **故障破坏了这个本意后，车辆实际发生了什么？**
   - 是”应该动却动不了”？还是”应该不动却自己动了”？
   - 是”应该加速却加速不了”？还是”应该减速却减速不了”？
   - 是”应该改变却卡住不变”？还是”应该不变却自己变了”？

### 常见逻辑陷阱

| 陷阱表现 | 为什么错 | 正确思考 |
|---------|---------|---------|
| 功能丧失 → 无法XX | “功能丧失”不等于”车辆无法移动” | 从车辆实际后果判断，而非功能状态 |
| 驻车失效 → 无法移动 | 驻车本意是”防止移动”，失效=无法防止=车辆动了 | 先问”这功能原本要让车辆做什么” |
| 纠纷”不能XX”和”非预期XX” | 词义混淆导致选错危害 | 从车辆后果反推，而非从功能症状反推 |

### 区分相似危害的方法

当面临两个相似选项时：

1. **无法纵向移动 vs 非预期的纵向移动**
   - 问：故障后车辆是自己动了，还是想动动不了？
   - 自己动了 → 非预期的纵向移动
   - 动不了 → 无法纵向移动

2. **非预期的减速 vs 非预期的失去减速**
   - 问：故障后车辆是在减速，还是减速能力丧失？
   - 在减速（不该减速时减速）→ 非预期的减速
   - 减速能力丧失 → 非预期的失去减速

3. **非预期的加速 vs 非预期的失去加速**
   - 问：故障后车辆是在加速，还是加速能力丧失？
   - 在加速（不该加速时加速）→ 非预期的加速
   - 加速能力丧失 → 非预期的失去加速

## 常见映射陷阱

- 除非驱动系统确实执行了相反运动方向，不要把功能方向错误写成车辆反向行驶。
- 不要把”功能失效””驾驶员无法操作””开关无响应”直接当作整车危害。
- 不要把传感器故障、软件故障、通信中断、诊断失败等内部原因写成危害。
- 一个 MF 只表达一个主要故障效应和一个主要整车危害，不要用”或/以及/同时”打包多个结果；只有在列举同一故障效应的等价机制时，才允许使用”或”。

## 质量要求

- `车辆级后果` 必须描述车辆实际运动或状态变化。
- `关键判断` 必须能区分相似危害选项。
- `选择理由` 不能只复述危害名称，必须说明为什么该危害比相似危害更合适。
- 最终描述必须能直接供 Stage3 生成 HARA 场景，不诱导 Stage3 混入相邻功能或内部原因。

验证命令和合并时机以 `SKILL.md` 为准。
