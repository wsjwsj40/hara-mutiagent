# Stage 2：整车危害映射方法

本文件只定义功能故障到整车危害的语义映射方法，不定义 JSON schema。输出结构以 `json-contracts.md` 为准。

## 目标

将当前 Stage1 单功能片段中的每个非 `nan` 功能故障，映射为一个车辆层面的危害。

## 基本规则

- 展开字段：`功能丧失`、`过大`、`过早`、`过小`、`过晚`、`非预期激活`、`卡滞`、`方向错误`。
- 一个非 `nan` 故障单元格对应一条 MF。
- `整车级危害` 必须逐字来自 `vehicle_hazards.json`。
- `故障描述` 描述功能级故障行为，不写内部原因，不提前写 Stage3 场景。
- 每条 MF 必须保留 Stage1 追溯信息，便于 Stage3 回到 Stage0 `detail_text`。

## 映射步骤

1. 识别功能本意：该功能原本要让车辆或系统实现什么？
2. 识别功能影响：故障后输出、状态、时机或方向发生什么异常？
3. 推导车辆级后果：车辆运动或状态实际会发生什么变化？
4. 区分关键对立面：例如“自己动了”还是“想动动不了”，“意外减速”还是“减速能力丧失”。
5. 从 `vehicle_hazards.json` 中选择最贴近的危害。
6. 在 `hazard_reasoning` 中记录功能影响、车辆级后果、关键判断和选择理由。

## 常见映射陷阱

- 驻车/保持失效导致车辆溜车时，通常是非预期移动，不是无法移动。
- 除非驱动系统确实执行了相反运动方向，不要把功能方向错误写成车辆反向行驶。
- 不要把“功能失效”“驾驶员无法操作”“开关无响应”直接当作整车危害。
- 不要把传感器故障、软件故障、通信中断、诊断失败等内部原因写成危害。
- 一个 MF 只表达一个主要故障效应和一个主要整车危害，不要用“或/以及/同时”打包多个结果。

## 质量要求

- `车辆级后果` 必须描述车辆实际运动或状态变化。
- `关键判断` 必须能区分相似危害选项。
- `选择理由` 不能只复述危害名称，必须说明为什么该危害比相似危害更合适。
- 最终描述必须能直接供 Stage3 生成 HARA 场景，不诱导 Stage3 混入相邻功能或内部原因。

验证命令和合并时机以 `SKILL.md` 为准。
