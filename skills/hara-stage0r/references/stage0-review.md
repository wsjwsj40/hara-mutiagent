# Stage 0R: 功能提取评审

目标：单独评审 Stage 0 的功能清单完整性、章节绑定和系统判断。

## 输入

- 原文抽取 JSON：`output/<run_id>_source_extraction.json`，或用户原始文本
- Stage 0 JSON：`output/<run_id>_stage0_function_mapping.json`

## 输出

- Review JSON：`output/<run_id>_stage0_review.json`
- 修正后的 Stage 0 JSON：仍写回 `output/<run_id>_stage0_function_mapping.json`

## 检查点

### 基础检查

1. 功能清单表中的每个功能都必须出现在 `function_mapping` 中，不能因为模型认为不是安全相关就删除。
2. 每个功能的 `detail_text` 应尽可能绑定后续章节说明；找不到详细章节时必须说明。
3. 分类、备注、章节号、证据块和系统线索是否准确。
4. 正文中新增但表格未列出的功能是否被合理保留为候选功能。
5. `非功能逻辑类`、`参数标定类`、`安全措施类` 是否被错误跳过。
6. 如果发现遗漏、误绑定或系统线索错误，修改 Stage 0 JSON，并在 review JSON 的 `fixes` 中说明。

### 子章节误提取检查（重点）

**检查是否把功能说明的子章节错误提取为独立功能**：

| 错误模式 | 识别方法 | 正确做法 |
|---------|---------|---------|
| 子章节提取为功能 | Function_ID 或名称包含"工作电源挡位"、"功能逻辑"、"触发条件"等子章节标题 | 这些不是独立功能，应合并到主功能的 detail_text |
| 过度拆分 | 一个主功能被拆成多个"子功能"，每个子功能对应一个子章节 | 保持功能完整性，所有子章节内容合并到主功能 |

**检查方法**：
1. 查看是否存在 `sub_functions` 字段，如果有，检查这些子功能是否真正是独立功能
2. 对比原文章节结构，子章节（如 `2.3.1.1`、`2.3.1.2`）通常是功能描述，不是独立功能
3. 主功能标题格式通常是 `...功能`，如 `2.3.1 静态开关拉起功能`
4. 如果子章节的内容是描述该功能的"工作条件"、"功能逻辑"、"触发条件"等，应合并到主功能

**正确的功能结构**：
- 一个主功能（如"静态开关拉起功能"）
- 其 `detail_text` 包含所有子章节内容（工作电源挡位、功能逻辑等）
- 不存在 `sub_functions` 字段（除非是真正的功能分解，如"自动驻车"分解为"施加"和"释放"）

最终 `review_log.result` 必须为 `pass`。
