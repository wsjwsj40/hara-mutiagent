# Stage 3A: 场景与危害事件生成

目标：为每个 MF 生成 10-20 条场景和对应的危害事件，不包含 SEC 评级。

## 使用方式

只在 Stage 3A 生成或修正场景时读取本文件。若只是检查 JSON 字段，优先读取 `references/json-contracts.md`；若是 SEC 评级，转到 Stage 3B，不要继续加载本文件。

## 需要读取

- 当前 MF 的 `stage3_context_<MF_ID>.json`
- `knowledge-base/automotive/hara/common/01-concepts.md`
- `knowledge-base/automotive/hara/common/operation_scenarios.json`
- `references/json-contracts.md`
- 对应系统知识库中与当前 MF 直接相关的功能说明

不要读取完整 Stage 0、完整 Stage 2、风险评估知识库或安全目标知识库。

## 处理方式

1. 每次只处理一个 MF。
2. 每个 MF 生成 10-20 条真实可信场景。
3. 本阶段只生成场景、危害事件、`scenario_reasoning` 和 `max_asil_planning`。
4. 不生成 S/E/C、ASIL、安全目标或安全状态。

## 场景生成核心目标

- 识别该 MF 的可信最高风险路径，而不是平均覆盖所有道路、天气和车辆状态。
- 先规划核心风险原型，再生成场景。
- 场景必须位于功能运行域内；只有“非预期激活/无请求输出/条件不满足仍输出”等故障可考虑运行域外危险输出，并必须解释为什么会发生。
- 不为追求高 ASIL 拼接互相矛盾的条件。

## 最大 ASIL 搜索规划（通用规则）

Stage 3A 不做最终 SEC 评级，但必须在生成场景前输出 `max_asil_planning`，说明如何寻找可信最高风险路径。

### 第一步：提取功能运行域

从 Stage3 context 的 `matched_functions.detail_text`、`operating_domain_hints` 和 MF 描述中提取：

- 功能触发方式：驾驶员请求、自动触发、系统条件触发。
- 功能作用对象：制动、转向、驱动、换挡、驻车、照明、警示、门锁、能量管理等。
- 功能输出会改变的车辆级状态：纵向运动、横向运动、稳定性、可见性、可通信性、约束/保持状态等。
- 运行边界：车速范围、车辆状态、道路/环境前置条件、是否需要驾驶员在环。
- 排除条件：功能明确不工作的状态或不应输出的状态。

### 第二步：识别风险驱动因子

按 S/E/C 三个方向找“可能推高风险”的事实，但不要在 Stage3A 给出最终评级。

| 方向 | 通用风险驱动因子 | 规划时要问的问题 |
|---|---|---|
| 可能提高 S | 高能量碰撞、弱势交通参与者、失去制动/转向/驱动控制、翻滚/坠落、火灾、电击、夹伤、误约束/失约束 | 该故障如何造成更严重的人身伤害？风险对象是谁？伤害能量从哪里来？ |
| 可能提高 E | 功能常用工况、高频车辆状态、常见道路/环境、目标用户经常遇到的运行域 | 这个场景是否在合理预期使用中经常出现？是否属于该功能真实运行域？ |
| 可能提高 C | 驾驶员不在控制回路、反应时间短、异常难以感知、动作反直觉、空间受限、低附着、其他交通参与者难以避让 | 谁能控制？能否及时感知？是否有足够时间、空间和可行动作？ |

### 第三步：生成候选场景原型

每个原型应描述一个“真实可发生的风险机制”，而不是枚举值组合。建议覆盖：

- 可信最高风险原型：最可能形成最高风险的真实组合。
- 接近最高风险原型：改变一个关键因子，观察风险边界。
- 常见高暴露原型：高频但不一定最严重的运行场景。
- 可控性边界原型：驾驶员在环/不在环、能感知/难感知、有空间/无空间等差异。
- 低风险或不适用边界原型：用于说明该 MF 的风险上限，数量要少。

同一核心原型最多保留 2 条。核心原型至少由这些要素定义：

- 故障输出或丧失的车辆级效果
- 车辆状态和速度范围
- 风险对象类型和位置
- 车辆实际运动方向或危险路径
- 驾驶员/相关人员是否在控制回路
- 决定暴露或可控性的关键条件

### 第四步：筛掉不可信组合

生成场景前必须剔除：

- 不在功能运行域内的请求相关故障场景。
- 为了提高风险而硬拼的极端条件。
- 只替换天气、道路名称、地标，但危害机制完全相同的重复场景。
- 风险对象不在车辆危险路径上的场景。
- 危害事件和车辆状态/车速/道路条件互相矛盾的场景。

### max_asil_planning 输出格式

`预期最大_ASIL` 是规划假设，不是最终评级；最终 SEC 由 Stage3B 判断，Stage3R 复核。

```json
{
  "max_asil_planning": {
    "高风险因素分析": [
      "可能提高S的因素：<基于当前MF的伤害能量/风险对象/危险输出>",
      "可能提高E的因素：<基于当前功能运行域的高频使用场景>",
      "可能提高C的因素：<基于感知、反应时间、控制回路和空间约束>"
    ],
    "规划的场景原型": [
      "原型1：<可信最高风险机制>（预期高风险原因：S/E/C 驱动因子）",
      "原型2：<接近最高风险或边界机制>（预期高风险原因：S/E/C 驱动因子）",
      "原型3：<常见高暴露或低风险边界机制>（用于验证风险上限）"
    ],
    "预期最大_ASIL": "<QM/A/B/C/D 或 待Stage3B确认>",
    "规划理由": "基于当前MF的功能运行域、整车危害和风险对象，说明为什么这些原型能覆盖可信最高风险路径。"
  }
}
```

### 规划验证

生成 10-20 条场景后，反向检查：

- 是否覆盖了 `max_asil_planning` 中的每类核心原型？
- 是否至少包含一个可信最高风险原型？
- 是否包含接近最高风险的边界场景，而不是只生成低风险场景？
- 是否存在重复原型、域外场景或互相矛盾条件？
- 是否为后续 Stage3B 判断 S/E/C 保留了足够事实，例如风险对象、车辆运动方向、速度/能量、暴露理由和可控性线索？

## 功能运行域约束

生成场景前必须从 Stage 0 `detail_text` 和 Stage3 context 提取功能运行域：

- 触发源、自动/手动
- 车速范围、车辆状态
- 驾驶员请求、前置条件
- 预期输出、排除条件

规则：

- 功能丧失、过大、过小、过早、过晚、卡滞、方向错误等请求相关故障必须发生在功能运行域内。
- 只有“非预期激活”“无请求输出”“不满足条件仍输出”等故障可以选择运行域外场景。
- 运行域外危险输出必须说明系统为什么在无请求或条件不满足时产生了危险输出。

## 场景字段约束

- 六大场景字段必须逐字来自 `operation_scenarios.json`。
- 库外描述只能写入 `附加条件`。
- 如果想表达场景库没有的概念，选择最接近的库内枚举，并把原词写入 `附加条件`。
- 对危害事件无影响的条件，在 `scenario_reasoning.场景条件相关性检查` 中写为 `不涉及，理由`；验证工具可据此规范化字段。

## 危害事件生成规则

填写每个场景的 `危害事件` 字段前，必须执行：

1. 确定车辆实际运动或危险输出：前向/后向/侧向运动、非预期加速、非预期减速、转向偏离、保持/约束失效、误动作等。
2. 确定风险对象位置：前方、后方、侧方、车内、车外、弱势交通参与者、其他车辆等。
3. 检查风险对象是否位于危险路径上。
4. 检查道路条件、车辆状态、车速和驾驶员是否在车上是否逻辑一致。

危害事件描述格式：

```text
<风险对象>在<场景条件下>因<车辆异常状态或危险输出>受到<伤害类型>风险
```

示例模板：

- `<风险对象>因车辆非预期运动/失去控制，发生碰撞受到伤害风险`
- `<车内人员/车外人员>因功能错误输出导致车辆状态异常，受到夹伤、碰撞、跌落、火灾或电击等伤害风险`

## 场景条件相关性检查

生成危害事件后，必须反向检查每个场景条件：

问题：这个条件的不同取值是否会改变危害事件的严重程度、出现概率或可控性？

- 如果不会：记录为 `不涉及，理由`
- 如果会：记录为 `相关，理由`

```json
"场景条件相关性检查": {
  "环境条件": "不涉及，白天/夜间不改变该危险输出和风险对象",
  "道路条件": "相关，低附着会改变制动距离和可控性"
}
```

判断标准：

| 条件 | 相关性判断 |
|---|---|
| 道路类型 | 是否影响风险对象类型、交通密度、逃避空间或功能运行域 |
| 道路条件 | 是否影响附着、坡度、碰撞能量、车辆稳定性 |
| 环境条件 | 是否影响能见度、路面摩擦、感知或暴露频率 |
| 车辆状态 | 是否与功能运行域和危险输出直接相关 |
| 车速 | 是否影响碰撞能量、反应时间或功能适用性 |
| 特殊要素 | 是否引入弱势对象、拥堵、施工、学校区域等关键风险条件 |

## 推理记录要求

每个场景必须包含 `scenario_reasoning`：

```json
{
  "场景规划理由": "本场景覆盖max_asil_planning中的某个风险原型，说明关键风险驱动因子",
  "危害事件推理": "故障导致车辆级状态如何变化，在该场景下风险对象为什么处于危险路径上",
  "场景条件相关性检查": {
    "道路类型": "相关/不涉及，理由",
    "道路条件": "相关/不涉及，理由",
    "环境条件": "相关/不涉及，理由",
    "车辆状态": "相关/不涉及，理由",
    "车速": "相关/不涉及，理由",
    "特殊要素": "相关/不涉及，理由"
  }
}
```

## JSON 写入规则

每次写入内容不得超过 200 行。如果 JSON 内容超过 200 行，分批写入，并确保最终拼接后是完整 UTF-8 JSON。

写入后验证：

```text
python tools/hara/check_stage_json.py --stage stage3a --json output/<run_id>_stage3a_<MF_ID>_scenarios.json --mf-id <MF_ID> --operation-scenarios knowledge-base/automotive/hara/common/operation_scenarios.json --min-scenarios 10 --max-scenarios 20 --fix
```
