## 本阶段文件

文件：`output/<run_id>_stage3a_<MF_ID>_scenarios.json`

只允许顶层 key：`meta`、`scenarios`、`max_asil_planning`、`review_log`。

### scenarios 固定列

- `List_No`
- `MF_ID`
- `故障描述`

- `道路类型`
- `道路条件`
- `环境条件`

### 场景字段约束（严格验证）

六大场景字段 `道路类型`、`道路条件`、`环境条件`、`车辆状态`、`车速（km/h）`、`特殊要素`：

**每个字段只能使用其对应场景库中的值**：

| JSON 字段 | operation_scenarios.json 键 | 常见值示例 |
|-----------|---------------------------|------------|
| `道路类型` | `道路类型` | 高速公路、城市主干路、乡村公路 |
| `道路条件` | `道路条件` | 直道、弯道、**泥泞路面**、冰雪路面、坑洼路面 |
| `环境条件` | `环境条件` | 日间、夜间、降雨(小/大/暴)、雾霾 |
| `车辆状态` | `车辆状态` | 启动起步、前向加速、转弯、泊车 |
| `车速（km/h）` | `车速（km/h）` | (0,10]、(30,60]、静止状态 |
| `特殊要素` | `特殊要素` | 与前车距离较近、行人数量(多)、学校区域、施工区域 |

**禁止跨字段使用值**：
- ❌ `道路条件` 的值（如泥泞路面、冰雪路面）不能放在 `特殊要素` 中
- ❌ `特殊要素` 的值（如学校区域、施工区域）不能放在 `道路条件` 中
- ❌ `环境条件` 的值（如夜间、降雨）不能放在其他字段中

**验证规则**：
1. 每个字段的值必须严格匹配 `operation_scenarios.json` 中对应键的值
2. 特殊允许值：`不涉及`、`ALL`
3. 库外描述只能写入 `附加条件`
4. **工具会自动检测字段误用并报告错误**

**在 `scenario_reasoning.场景条件相关性检查` 中记录相关性判断**

### 自动修改机制

工具 `check_stage_json.py --fix` 会根据推理自动将原字段值修改为 `不涉及`。

你只需要在推理中正确判断：
- 如果**不影响**：记录为 `"不涉及，理由"`
- 如果**影响**：记录为 `"相关，理由"`

### 判断标准

条件设置为 `不涉及` 的标准：
- 该条件的不同取值**不会影响**危害事件的严重程度
- 该条件的不同取值**不会影响**危害事件的发生概率
- 该条件与危害事件的因果链**无关**

判断方法：问"改变这个条件的值，危害事件会变得更严重或更可能发生吗？"如果不会，则为 `不涉及`。

### max_asil_planning 结构（必须包含）

用于记录该 MF 的最大 ASIL 搜索规划：

```json
{
  "max_asil_planning": {
    "高风险因素分析": [
      "可能提高S的因素：<当前MF可能造成的人身伤害机制>",
      "可能提高E的因素：<当前功能运行域内的高频使用场景>",
      "可能提高C的因素：<感知、反应时间、控制回路或空间约束>"
    ],
    "规划的场景原型": [
      "原型1：<可信最高风险机制>（预期高风险原因：S/E/C驱动因子）",
      "原型2：<接近最高风险或边界机制>（预期高风险原因：S/E/C驱动因子）",
      "原型3：<常见高暴露或低风险边界机制>（用于验证风险上限）"
    ],
    "预期最大_ASIL": "QM/A/B/C/D 或 待Stage3B确认",
    "规划理由": "基于当前MF的功能运行域、整车危害和风险对象，说明为什么这些原型能覆盖可信最高风险路径"
  }
}
```

**字段说明**：
- `高风险因素分析`：列出当前 MF 中可能推高 S/E/C 的事实，不要套用固定案例。
- `规划的场景原型`：列出规划生成的核心风险机制，不能只是枚举字段组合。
- `预期最大_ASIL`：规划假设，不是最终评级；最终 SEC 由 Stage3B 判断。
- `规划理由`：说明为什么这些场景能覆盖最大 ASIL

这个规划记录用于 Stage3R 评审时验证是否正确覆盖了最大 ASIL 搜索路径。
