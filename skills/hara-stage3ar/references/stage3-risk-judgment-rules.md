# Stage 3 风险判断与场景合理性规则

本文件是执行规则，不是标准知识库。它用于约束 Stage3A 场景生成、Stage3AR 场景评审和 Stage3BR SEC 复核，避免模型为了抬高或降低 ASIL 生成不现实、不自洽的场景。

## 使用方式

只在 Stage3AR/Stage3BR 发现运动方向、风险对象、S/E/C 边界或场景独立性争议时读取本文件。

## 危害事件生成强制检查清单

在填写每个场景的**危害事件**字段前，必须按以下步骤检查：

**第一步：确定车辆实际运动方向**
- 道路条件是什么？（上坡/下坡/平路/弯道）
- 故障会导致车辆如何运动？（前向滑移/向后溜车/侧向滑移/横摆）
- 重力/惯性如何影响车辆？

**第二步：确定风险对象位置**
- 如果车辆向后运动 → 风险对象在车辆**后方/坡下方向**
- 如果车辆前向运动 → 风险对象在车辆**前方/坡下方向**
- 上坡向后溜车 → 后方/坡下，**不是前方**
- 下坡向前滑移 → 前方/坡下

**第三步：选择正确术语**
- 车辆向后运动撞击后方车辆 → "本车倒溜撞击后方车辆"
- **禁止**：上坡向后溜车时写"撞击前车/前方车辆"

**常见错误对照**：
| 场景 | 错误写法 | 正确写法 |
|-----|---------|---------|
| 上坡向后溜车 | 撞击前车 | 撞击后方车辆/坡下车辆 |
| 下坡向前滑移 | 撞击后车 | 撞击前方车辆/坡下车辆 |
| 本车向后倒退 | 追尾后车 | 倒退撞击后方车辆 |

## 总体原则

- **场景生成目标**：每个 MF 应生成 10-20 条场景，目标是**识别该 MF 的可信最大 ASIL 等级**。场景不是随机发散的使用场景列表，而是围绕该 MF 可能达到最高风险的真实场景进行分析。
- **先找高风险场景，再按事实评级**：场景选择阶段应主动寻找可能推高 ASIL 的真实因素组合；S/E/C 评级阶段必须严格回到事实和定义，不得为了得到高 ASIL 而抬高等级。
- S/E/C 不是为了凑高 ASIL 或降低 ASIL 的主观调节项，必须由危害事件和具体场景事实支撑。
- 场景应优先选择真实车辆使用中常见或可合理发生的组合。不要使用现实中几乎不会发生的场景来抬高 ASIL。
- **避免过度离散**：场景应聚焦于有意义的 S/E/C 组合变化。不要机械组合道路类型、道路条件、环境条件产生大量实质相同的场景。
- **SEC等级与解释必须一致**：所选等级必须与等级定义匹配，解释不能自相矛盾。例如不能选 E2（低概率）却解释为"常见场景"。
- 常见场景应占主要比例，极端场景应是合理最坏场景，而不是无概率的想象场景。
- 对当前故障无影响的条件不要作为风险放大或降级依据。夜间、雨雪、道路类型等条件只有在改变感知、附着、制动距离、交通流、暴露频率或逃避空间时，才可影响 S/E/C。
- 雨、雪、夜间、盘山路不能单独把 S 提升到 S3。只有当场景明确说明更高碰撞速度、更长滑移距离、坠落/翻滚、近距离弱势交通参与者且无逃避空间等事实时，才可提高 S。
- 不要把 `S3+E1+C3` 写成 D。后缀求和为 7，结果是 A。所有 ASIL 文字说明都必须遵守 `score = S后缀 + E后缀 + C后缀`。

## 最大 ASIL 寻找策略

- 对每个 MF 先列出 3 类风险驱动因素：可能提高 S 的伤害能量或受害对象，可能提高 E 的真实暴露场景，可能提高 C 的难控制条件。
- 优先组合“同一真实场景中可以同时成立”的高风险因素。例如高伤害能量、常见运行工况、反应时间短、风险对象处在危险路径上等；不能把互相矛盾或不属于功能运行域的因素硬拼在一起。
- 如果最高 S 场景只在极低频特殊条件下出现，E 必须按特殊条件真实评估，不能为了高 ASIL 把 E 写高。
- 如果高 E 场景只会造成低能量、可控后果，S/C 必须按低风险事实评估，不能为了高 ASIL 写成严重伤害或不可控。
- 每组 10-20 条场景中，应优先覆盖可信最高风险原型和接近最高风险的变体；低风险场景只用于说明边界、对比或证明该 MF 最高 ASIL 的限制因素。
- 当分析后发现该 MF 的可信最高 ASIL 只能达到 QM/A/B，也应如实输出，不得为了“找最高”而制造不现实的 D 场景。

## 场景独立性规则

- “独立场景”必须改变核心风险原型，而不只是改环境词。
- 核心风险原型至少包括：故障输出/整车危害、道路条件、车辆状态、车速、驾驶员是否在车上、主要风险对象、车辆相对运动方向。
- 同一 MF 中，同一核心风险原型最多出现 2 次。若第 3 次只是在日间/夜间/雨雪/盘山路/城市支路之间替换，应合并或替换。
- 环境条件只有在改变可见性、制动距离、附着能力、交通参与者暴露或可控性时，才可作为独立场景依据，并且必须在 `附加条件`、`危害事件`、`S/E/C解释` 中具体说明。

## S 判断规则

- S 关注“可能造成的人身伤害严重度”，不是故障听起来是否严重。
- 先列出每类有风险人员，再分别判断其可能后果和 S 等级，最终 `Severity 'S'` 取最高等级。
- 低速轻微剐蹭、轻微惊吓、短距离低能量碰撞通常不应直接判 S3，可考虑 S1/S2。
- 高速追尾、多车碰撞、对向碰撞、车辆失控撞击弱势交通参与者、坡道溜车撞击近距离行人等才重点考虑 S3。
- “可能撞击行人”不必然等于 S3；应结合车速、坡度、距离、行人位置、逃避空间和碰撞能量判断。低速、空间充足时可为 S2；近距离、无逃避空间或车辆有明显动能时可为 S3。
- 从静止或低速开始的危险运动，通常先按低速/短距离风险判断。若评为 S3，必须说明车辆如何获得足够动能，或出现坠落、翻滚、挤压、近距离弱势交通参与者且无逃避空间等严重后果条件。
- 速度可参考 `04-risk_assessment-s.md` 中的车车碰撞和弱势交通参与者 S 等级速度表。

## E 判断规则

- E 关注运行场景出现频率，不关注故障发生概率；不要因为故障严重就降低 E。
- 城市道路、停车场、坡道停车、起步、熄火停车、开关操作、正常天气等普通用户会周期性遇到的场景，通常不应过度保守地评为 E0/E1。
- 维修车间、展台、拖车模式、特殊测试、极端天气、越野或山区陡坡等较少出现但合理的场景，通常考虑 E1/E2。
- 高频场景可考虑 E3/E4，但必须与该功能适用工况一致。只在某一运行域内工作的功能，不应为了提高 E 使用运行域外场景；除非故障类型本身是非预期激活或无请求输出，并且能解释域外危险输出的发生逻辑。
- 极端天气和盘山路等条件不能自动把 E 降到 E1；应结合目标用户和功能适用工况判断。如果只是普通雨天城市道路，通常不应按极低概率处理。
- `E-解释` 必须包含“所选 E 等级定义 + 当前场景出现频率理由”。可引用 `04-risk_assessment-e.md` 中的持续时间 d 和频率 f 表。

## C 判断规则

- C 关注相关人员避免伤害的现实能力。驾驶员是否在车上、是否能感知、是否有足够反应时间和可行动作，是核心判断点。
- 驾驶员离车、车辆已进入非预期运动、行人位于车辆运动方向近距离区域时，通常难以控制，可考虑 C3。
- 驾驶员在车内且风险明显、有告警、有足够距离和时间进行制动、转向或重新操作时，可考虑 C1/C2。
- 对行驶中突发制动、突发转向、低附着弯道、后车距离近等快速演化事件，通常可控性较低。
- 告警、提示、DTC 只能作为有助于感知的因素，不能自动把 C3 降为 C1。
- `C-解释` 必须包含“所选 C 等级定义 + 当前场景可控性理由”。

## 运动方向与风险对象

- 危害事件中的风险对象必须位于合理的危险路径上。
- 坡道上保持、制动、驱动、换挡或其他会影响车辆纵向运动的故障，可能使车辆沿坡下方向滑移。风险对象应位于坡下方或车辆实际滑移方向，而不是机械写成后方、前方或某一固定位置。
- 上坡或下坡场景要先判断车辆实际运动方向，再确定风险对象位置；例如车辆倒退下滑时，风险对象通常在车辆后方/坡下方，车辆前向下滑时，风险对象通常在车辆前方/坡下方。
- 行驶中非预期减速主要风险是后车追尾本车；不要写成本车追尾前车，除非场景包含前车距离过近且本车非预期加速或失去减速。
- 非预期加速或失去减速时，主要风险可能是本车追尾前车、冲出路口或撞击前方行人。
- 非预期横摆或左右制动力不一致时，风险通常与车道偏离、侧向碰撞、对向来车或相邻车道车辆相关。

## 碰撞术语与方位规则

- 写危害事件前必须按四步检查：本车实际运动方向 -> 风险对象相对位置 -> 碰撞接触关系 -> 术语是否匹配。
- “追尾”只用于同向前后车辆的前后碰撞关系：
  - 本车前向运动撞到前方车辆：可写“本车追尾前车”或“本车撞击前方车辆”。
  - 后方车辆撞到本车尾部：可写“后车追尾本车”或“本车被后车追尾”。
  - 本车向后倒退、后溜或倒溜撞到后方车辆：必须写“本车倒退/倒溜撞击后方车辆”，不得写“本车追尾后车”“车辆向后溜车追尾后车”或“与后车发生追尾”。
- 上坡停车或堵车等待时，如果保持/制动/驻车能力不足导致车辆向后溜车，风险对象通常在本车后方或坡下方向；危害事件应写“向后溜车撞击后方车辆/行人/障碍物”，不能写成“追尾后车”。
- 下坡停车或等红灯时，如果车辆沿下坡方向前向滑移，风险对象通常在前方或坡下方向；危害事件应写“前向滑移撞击前方车辆/行人/障碍物”。只有明确车辆朝向与坡向相反并向后滑移时，才可写后方对象，并必须在 `附加条件` 中说明车辆朝向。
- “后车追尾本车”和“本车倒溜撞击后车”是两种不同事件：前者是后方车辆主动撞上本车，后者是本车向后运动撞上后方车辆。不要混用。

## 解释字段写法

- `E-解释` 必须包含”所选 E 等级定义 + 当前场景出现频率理由”。
- `C-解释` 必须包含”所选 C 等级定义 + 当前场景可控性理由”。
- `可能的后果('S'的理由)` 必须逐个对应 `有风险的人员` 中列出的对象，并给出各自可能 S 等级，最终 `Severity 'S'` 取最高等级。

**格式示例**：

```json
“E-解释”: “E1：极低概率（非常低的发生概率，场景极少出现，通常小于 0.4 小时/年）。本场景为坡道停车且驾驶员离开，属于低频场景，评为E1。”
```

```json
“C-解释”: “C1：简单可控（大多数驾驶员或相关人员能够通过直接、常规的反应避免伤害，有足够时间反应）。本场景中后方驾驶员或行人有一定反应时间可以避让，评为C1。”
```

**错误示例**（缺少等级定义）：

```json
“E-解释”: “E1：停车离开后车辆状态变化，驾驶员不在车内，无法及时发现。坡道停车是常见场景。”
```

```json
“C-解释”: “C1：后方驾驶员或行人有一定反应时间，但高速溜车时难以躲避”
```

**正确写法**：先写出该等级的定义（S/E/C 分别来自 `04-risk_assessment-s.md`、`04-risk_assessment-e.md`、`04-risk_assessment-c.md`），再说明本场景为什么评这个等级。
