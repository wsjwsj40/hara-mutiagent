# Stage 1R：语义评审方法

## 定位

Stage1R 评审的是 Stage1 单功能片段的语义质量，不评审 JSON schema。

- 结构、字段、行数、`field_reasoning` 机械一致性由 `check_stage_json.py --stage stage1_slice --fix` 处理。
- Stage1R review 文件只做人工审查留痕，不作为 Stage2 输入。
- 最终硬门禁是修正后的 Stage1 单功能片段，以及合并后的最终 Stage1 JSON。

## Review 留痕建议

Review 文件必须是合法 JSON，并能识别当前 `Function_ID`。推荐包含：

```json
{
  "meta": {
    "function_id": "<Function_ID>",
    "target_file": "output/<run_id>_stage1_<Function_ID>_derive_mf.json"
  },
  "overall_result": "pass/failed",
  "issues": [],
  "fixes": [],
  "notes": ""
}
```

这只是建议结构，不是严格 schema。`merge_stage1_review.py` 会优先读取 `meta.function_id`；如果没有，则尝试从文件名 `<run_id>_stage1_<Function_ID>_review.json` 推断。

## 评审顺序

1. 确认当前 Stage1 单功能片段已经通过 `stage1_slice --fix`。
2. 先看校验 warnings，定位可能的多故障打包、字段不一致或可疑描述。
3. 对 `nan` 字段重新推理，确认 `是否适用` 是否真的为否。
4. 对非 `nan` 字段区分 `是否适用` 和 `是否有安全风险`：`是否适用` 只判断该故障类型是否能构成当前功能的功能级异常；适用但无安全风险的故障应留在 Stage1，不应进入 Stage2。
5. 重点复核 `过小`、`方向错误`、功能边界和故障描述质量。
6. 发现语义错误时，直接修正 Stage1 单功能片段，并在 review 文件中记录依据。
7. 修改后重新运行 `stage1_slice --fix`。

## 逐字段方法

对每个故障字段问三个问题：

- 功能本质是什么：力、扭矩、压力、状态、信息、时机、方向，还是保持/约束/保护能力？
- 当前故障类型是否真的适用：异常会不会导致功能输出缺失、过量、过早、过晚、反向、卡滞或非预期？
- 最终描述是否停留在功能级异常：不要写内部原因、软件故障、诊断失败，也不要提前写成整车危害。

## 重点复核项

- `nan` 字段：必须有真实的不适用理由，不能只是因为模型“不确定”。
- 适用但无安全风险：故障描述必须保留在 Stage1，同时 `是否有安全风险` 标为 `否`，供 Stage2 过滤。
- `过大` 非安全相关边界：如果过大只导致部件过载、磨损、寿命降低或设备损坏，需保留 `derive_mf` 故障描述，并在 `derive_mf` 描述中明确不直接造成人身伤害风险；推理中的 `是否适用` 应为 `是`，`是否有安全风险` 应为 `否`。
- `过小`：保持、约束、保护、制动、力、压力、扭矩类功能通常需要重点怀疑，输出不足往往有安全风险。
- `卡滞` 目标状态一致性：对"请求 A/执行 A/切换到 A"的功能，卡滞应描述 A 无法建立、无法切换到 A、执行过程中无法完成到 A，或保持在 A 的对立/初始状态；不要写成 A 成功后无法切换到相邻或相反功能 B。
- `方向错误`：先找当前功能的"请求/条件 -> 预期动作或目标状态"，再检查同一执行对象、控制量或状态机是否存在相反动作/状态；不能只因子功能名称像单方向动作就判为不适用。
- 边界混入：故障描述不能混入相邻功能动作。
- 原因混入：故障字段描述的是异常行为，不是传感器故障、软件 bug、通信中断等内部原因。
- 危害前置：Stage1 只写功能故障，不写 Stage2 的整车危害结论。
- 多效应打包：一个单元格不要用“或/以及/同时”塞多个不同故障效应。

## `nan` 复核模板

```text
字段：<故障字段>
当前值：nan
功能输出/状态/时机：<当前功能本质>
是否适用：<是/否>
是否有安全风险：<是/否>
理由：<为什么该故障类型适用/不适用，以及适用时是否需要进入 Stage2>
结论：<保持 nan / 修改为具体故障描述>
```

## 常见错误

| 错误类型 | 表现 | 修正方向 |
|---|---|---|
| 幅值和时机混淆 | 把输出过大写成过早，或把过晚写成过小 | 先判断异常是幅值问题还是时序问题 |
| 过大安全边界缺失 | 输出强度过大只写部件损坏，却未标明不直接造成人身伤害风险 | 保留故障描述，补充非人身伤害边界，并将 `是否适用` 设为 `是`、`是否有安全风险` 设为 `否` |
| 过小遗漏 | 保持/保护/制动功能的过小填 `nan` | 重新分析输出不足是否导致安全风险 |
| 卡滞目标反向 | 请求 A/切换到 A 的功能写成"A 已完成后无法退出到 B" | 改为"A 无法建立、无法切换到 A，或执行 A 过程中无法完成状态切换" |
| 方向错误遗漏 | 存在互斥对立动作/状态的功能把方向错误填 `nan` | 分析同一对象、控制量或状态机中是否存在相反动作或相反状态 |
| 安全风险替代适用性 | 适用但无安全风险的故障被改成 `nan` | `是否适用=是` 时保留 Stage1 故障描述，`是否有安全风险=否` 只影响 Stage2 |
| 机械拼接 | 只有功能名加故障词 | 改成具体异常行为 |
| 相邻功能混入 | 当前功能字段写了其他功能的动作 | 限定在当前 Function_ID 的 `detail_text` |
| 危害前置 | 写成车辆碰撞、人员受伤等结论 | 留到 Stage2，Stage1 只写功能故障 |

## 修正原则

- 只修正证据充分的问题。
- 如果只是格式、空值、`是否适用` 与字段值机械不一致，先用 `check_stage_json.py --fix`。
- 如果语义判断错误，修改 Stage1 单功能片段，并在 review 留痕中记录 `target`、`change`、`reason`。
- 修改后的 Stage1 单功能片段必须重新通过 `stage1_slice --fix`。
