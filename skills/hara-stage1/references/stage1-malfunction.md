# Stage 1: 功能故障生成

目标：对 Stage 0 的每个功能生成一行 DeriveMF。不能跳过任何 Stage 0 功能。

## 文档定位

本文件只定义功能故障判断方法，不定义输出 JSON 结构。

- 输出字段、顶层 key、`field_reasoning` 结构和一致性约束以 `references/json-contracts.md` 为唯一来源。
- 知识库读取边界由 `skills/hara-stage1/SKILL.md` 控制；系统知识和故障定义仅在判断不清时按需参考。

## 规则

1. `derive_mf` 行数必须等于 Stage 0 `function_mapping` 行数。
2. 每个功能只能生成一行。
3. 按 `references/json-contracts.md` 中约定的故障字段逐项判断，不要新增、合并或改名字段。

4. `过大`、`过早`、`过小`、`过晚` 必须拆开判断。幅值问题和时序问题不能合并。

5. 不适用的故障模式填 `nan`，但必须是结合功能逻辑分析后确认不适用。适用但暂不形成安全相关危害的故障模式仍要填写故障描述，后续 Stage2 会按 `是否有安全风险` 过滤。

6. 每个故障单元格只能描述一个清晰的故障效应。不要用“或/以及/同时”把两个相反状态、两个相邻功能或两个不同整车危害合并在一个单元格中；否则 Stage 2 只能生成一个 MF，Stage 3 会把多个故障混在一起。

7. **通用判断方法**（对每个功能逐字段判断）：

   **核心原则：先判断"这个故障类型在当前功能边界内是否适用"，再判断"适用后的故障是否有安全风险"。**

   - `是否适用=是`：当前功能确实可能出现该类异常行为，Stage1 必须写出功能级故障描述。
   - `是否适用=否`：当前功能边界、输出形式或触发机理不支持该类异常，Stage1 才能填 `nan`。
   - `是否有安全风险=是`：该适用故障需要进入 Stage2 做整车危害映射。
   - `是否有安全风险=否`：该适用故障只保留在 Stage1，不进入 Stage2；不能因此把 Stage1 字段改成 `nan`。

   **重要区分：输入动作 vs 功能输出**
   - 输入动作（驾驶员拉开关、按按钮）可能是离散的
   - 但功能输出（制动力、夹紧力、扭矩、角度）可能是连续的
   - "过大/过小"判断的是**功能输出的效果**，不是**输入的动作**

   - **功能丧失**：功能完全不执行或无输出。先判断"功能完全不工作"是否是当前功能的有效失效行为；再判断是否有安全风险。几乎所有功能都适用。

   - **过大**：判断"功能输出的强度/幅值超出设计上限"是否是当前功能的有效异常。
     - 判断依据：功能输出是什么？输出是否存在可度量的强度、幅值、持续时间、频率或动作速度？输出过大是否会造成人身伤害相关危害？
     - 示例：夹紧力、制动力等输出过大不一定有危害，需结合功能边界判断；转向角度、驱动扭矩、制动力矩等输出过大可能导致失控或碰撞，需填写。
     - 如果异常适用但主要后果是部件过载、磨损、寿命降低或设备损坏，且不直接形成乘员/行人人身伤害风险，则 `derive_mf` 仍填写故障描述，但描述中要点明"不直接造成人身伤害风险"或等价边界，同时 `是否有安全风险=否`。

   - **过小**：判断"功能输出的强度/幅值低于设计下限"是否是当前功能的有效异常。
     - 判断依据：功能输出是什么？输出过小是否会导致预期控制、保持、提醒、约束或保护能力不足？
     - 示例：夹紧力或保持力不足可能导致车辆移动，制动力不足可能导致制动距离延长，警告强度不足可能导致驾驶员未及时感知。
     - 重点：如果功能涉及"保持、约束、保护、制动、锁止"，输出过小几乎总是适用且安全相关，不应轻易填 `nan`。

   - **过早**：判断"功能的执行时机早于设计意图"是否是当前功能的有效异常。
     - 驾驶员即时触发的离散行为（拉起开关、按下按钮）：不存在"提前执行"的概念，可填 `nan`
     - 自动功能（自动驻车、自动释放）：在条件未满足时提前执行可能导致危害，需填写
     - 判断依据：功能的触发时机由谁决定？驾驶员即时触发则不适用，系统自动判断则可能适用

   - **过晚**：判断"功能的执行时机晚于设计意图"是否是当前功能的有效异常。
     - 响应延迟导致驾驶员意图无法及时实现，需填写

   - **非预期激活**：判断"无驾驶员请求或不满足条件时自动激活"是否是当前功能的有效异常。
     - 即使功能由驾驶员主动触发，也要考虑系统故障导致的意外激活。几乎所有功能都适用。

   - **卡滞**：判断"功能状态无法切换"是否是当前功能的有效异常。
     - 描述的是功能状态（拉起/释放、开启/关闭、夹紧/松开）无法切换，不是输入开关的机械故障
     - 判断依据：功能是否有状态切换？卡滞在某状态是否有危害？
     - 必须限制在当前功能边界内。卡滞描述应围绕当前功能的状态迁移或执行过程，不要把相邻功能的故障写入当前功能；除非原功能文档明确当前功能包含该相邻行为。
     - 卡滞方向必须跟当前功能的目标状态一致。对"请求 A/执行 A/切换到 A"的功能，卡滞应描述 A 无法建立、无法切换到 A、执行过程中无法完成到 A，或保持在 A 的对立/初始状态；不要写成 A 已经完成后无法切换到相邻或相反功能 B，除非当前 Function_ID 明确包含退出 A/执行 B。
     - 对任何目标状态类功能，先抽取当前 Function_ID 的目标状态 A 和相反/相邻状态 B；如果当前功能只负责进入 A，则卡滞不能描述为进入 A 后无法退出到 B，这属于另一功能边界。

   - **方向错误**：判断"功能执行了与请求或设计意图相反的状态、动作、控制方向或目标方向"是否是当前功能的有效异常。
     - 判断依据 1：从功能定义中抽取"触发/请求/条件 -> 预期动作或目标状态"的配对关系。
     - 判断依据 2：检查同一执行对象、同一控制量或同一状态机中是否存在互斥对立面，例如 A/B、开启/关闭、夹紧/松开、左/右、加/减、升/降、前进/后退。
     - 判断依据 3：如果当前子功能只命名了其中一个方向，但系统或功能边界中存在同一对象的相反动作/状态，则仍要判断"请求 A 却执行相反 B"；不能因为当前子功能名称是单向表达就直接判为不适用。
     - 适用时，Stage1 故障描述应写成功能级反向行为，例如"请求 A 时执行相反 B"或"目标方向应为 A 却输出 B"，不要直接跳到车辆碰撞、人员受伤等整车危害。
     - 不适用情况：只有数值大小变化而无正反/互斥方向的纯幅值控制；只有单次通知、记录、发送等不可产生相反动作或相反状态的功能；功能文档和系统边界均找不到同一对象的对立状态。

8. **推理过程要求**：必须采用**两阶段推理**方式，确保推理过程是得出答案的前提，而不是事后补充。

   **阶段 1：推理阶段**
   先为每个故障字段生成推理记录，此时尚不确定最终答案。推理记录包含：
   - 功能输出/状态/时机分析
   - 异常情况定义
   - 后果推导
   - 是否适用的判断
   - 是否有安全风险的判断
   - **暂不包含最终结论或故障描述**

   **阶段 2：结论阶段**
   基于阶段 1 的推理记录，得出最终结论：
   - 如果 `是否适用=是` → 生成具体故障描述
   - 如果 `是否适用=否` → 填写 `nan`
   - 如果 `是否适用=是` 且 `是否有安全风险=否` → 故障描述保留在 Stage1，但 Stage2 不生成 MF

   推理记录的输出结构以 `references/json-contracts.md` 中的 `field_reasoning` Schema 为准。

   **重要约束**：
   - `推理.是否适用` 直接决定 Stage1 最终值：`是` → 填写故障描述，`否` → 填写 `nan`
   - `推理.是否有安全风险` 只决定是否进入 Stage2；不能用它覆盖 Stage1 的故障适用性
   - `是否有安全风险=是` 时，`是否适用` 必须为 `是`
   - 推理过程必须独立于最终故障描述，不能是先有答案再补充推理
   - `是否适用` 必须是明确的 `是` 或 `否`，不能是模糊描述
   - `是否有安全风险` 必须是明确的 `是` 或 `否`，不能是模糊描述

9. 必须参考 Stage 0 的 `detail_text`、功能设计意图、系统知识和故障类型定义，不要模板化拼接。

写入和验证命令以 `skills/hara-stage1/SKILL.md` 为准。
