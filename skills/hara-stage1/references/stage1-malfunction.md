# Stage 1: 功能故障生成

目标：对 Stage 0 的每个功能生成一行 DeriveMF。不能跳过任何 Stage 0 功能。

## 需要读取

- `knowledge-base/automotive/hara/common/01-concepts.md`
- 对应系统知识库：`knowledge-base/automotive/hara/systems/<system>/01-basic_information.md`
- `knowledge-base/automotive/hara/common/02-malfuntioning_behavior.md`
- `references/json-contracts.md`

## 规则

1. `derive_mf` 行数必须等于 Stage 0 `function_mapping` 行数。
2. 每个功能只能生成一行。
3. 固定字段：`No.`、`子功能`、`功能丧失`、`过大`、`过早`、`过小`、`过晚`、`非预期激活`、`卡滞`、`方向错误`。

4. `过大`、`过早`、`过小`、`过晚` 必须拆开判断。幅值问题和时序问题不能合并。

5. 不适用的故障模式填 `nan`，但必须是结合功能逻辑分析后确认不适用。

6. 每个故障单元格只能描述一个清晰的故障效应。不要用“或/以及/同时”把两个相反状态、两个相邻功能或两个不同整车危害合并在一个单元格中；否则 Stage 2 只能生成一个 MF，Stage 3 会把多个故障混在一起。

7. **通用判断方法**（对每个功能逐字段判断）：

   **核心原则：不是判断"有没有这个属性"，而是判断"这个异常方向是否会产生安全相关危害"。**

   **重要区分：输入动作 vs 功能输出**
   - 输入动作（驾驶员拉开关、按按钮）可能是离散的
   - 但功能输出（制动力、夹紧力、扭矩、角度）可能是连续的
   - "过大/过小"判断的是**功能输出的效果**，不是**输入的动作**

   - **功能丧失**：功能完全不执行或无输出。判断"功能完全不工作"是否会导致危害。几乎所有功能都适用。

   - **过大**：判断"功能输出的强度/幅值超出设计上限"是否会导致危害。
     - 判断依据：功能输出是什么？输出过大是否会造成人身伤害相关危害？
     - 示例：夹紧力、制动力等输出过大不一定有危害，需结合功能边界判断；转向角度、驱动扭矩、制动力矩等输出过大可能导致失控或碰撞，需填写。

   - **过小**：判断"功能输出的强度/幅值低于设计下限"是否会导致危害。
     - 判断依据：功能输出是什么？输出过小是否会导致预期控制、保持、提醒、约束或保护能力不足？
     - 示例：夹紧力或保持力不足可能导致车辆移动，制动力不足可能导致制动距离延长，警告强度不足可能导致驾驶员未及时感知。
     - 重点：如果功能涉及"保持、约束、保护、制动、锁止"，输出过小几乎总是适用且安全相关，不应轻易填 `nan`。

   - **过早**：判断"功能的执行时机早于设计意图"是否会导致危害。
     - 驾驶员即时触发的离散行为（拉起开关、按下按钮）：不存在"提前执行"的概念，可填 `nan`
     - 自动功能（自动驻车、自动释放）：在条件未满足时提前执行可能导致危害，需填写
     - 判断依据：功能的触发时机由谁决定？驾驶员即时触发则不适用，系统自动判断则可能适用

   - **过晚**：判断"功能的执行时机晚于设计意图"是否会导致危害。
     - 响应延迟导致驾驶员意图无法及时实现，需填写

   - **非预期激活**：判断"无驾驶员请求或不满足条件时自动激活"是否会导致危害。
     - 即使功能由驾驶员主动触发，也要考虑系统故障导致的意外激活。几乎所有功能都适用。

   - **卡滞**：判断"功能状态无法切换"是否会导致危害。
     - 描述的是功能状态（拉起/释放、开启/关闭、夹紧/松开）无法切换，不是输入开关的机械故障
     - 判断依据：功能是否有状态切换？卡滞在某状态是否有危害？
     - 必须限制在当前功能边界内。卡滞描述应围绕当前功能的状态迁移或执行过程，不要把相邻功能的故障写入当前功能；除非原功能文档明确当前功能包含该相邻行为。

   - **方向错误**：判断"功能执行了与请求相反的状态或动作"是否会导致危害。
     - 判断依据：分析功能定义，提取"请求→预期动作"的配对关系。如果功能有两个或多个互斥的相反状态/动作（如夹紧/释放、前进/后退、左转/右转），且"请求A却执行非A"会造成人身伤害，则填写。
     - 不适用情况：纯幅值控制（只有大小/强弱变化，无相反方向，如音量、亮度）；单向功能（只有一个执行方向，如一次性触发、单向发送）。

8. **推理过程要求**：必须采用**两阶段推理**方式，确保推理过程是得出答案的前提，而不是事后补充。

   **阶段 1：推理阶段**
   先为每个故障字段生成推理记录，此时尚不确定最终答案。推理记录包含：
   - 功能输出/状态/时机分析
   - 异常情况定义
   - 后果推导
   - 是否有安全风险的判断
   - **暂不包含最终结论或故障描述**

   **阶段 2：结论阶段**
   基于阶段 1 的推理记录，得出最终结论：
   - 如果推理显示有安全风险 → 生成具体故障描述
   - 如果推理显示无安全风险或不适用 → 填写 `nan`

   **输出结构**：
   ```json
   {
     "derive_mf": [...],
     "field_reasoning": [
       {
         "row": 1,
         "子功能": "静态开关拉起",
         "字段推理": [
           {
             "字段": "过小",
             "推理": {
               "功能输出": "夹紧力（用于保持车辆静止）",
               "异常情况": "夹紧力低于设计下限",
               "后果": "无法有效保持车辆静止，车辆可能溜车",
               "是否有安全风险": "是"
             }
           },
           {
             "字段": "过早",
             "推理": {
               "功能输出": "驾驶员即时拉起开关",
               "异常情况": "在拉起开关之前执行",
               "后果": "不存在提前执行的概念",
               "是否有安全风险": "否"
             }
           }
         ]
       }
     ]
   }
   ```

   **重要约束**：
   - `推理.是否有安全风险` 直接决定最终值：`是` → 填写故障描述，`否` → 填写 `nan`
   - 推理过程必须独立于最终故障描述，不能是先有答案再补充推理
   - `是否有安全风险` 必须是明确的 `是` 或 `否`，不能是模糊描述

9. 必须参考 Stage 0 的 `detail_text`、功能设计意图、系统知识和故障类型定义，不要模板化拼接。

写入后运行。`tools/hara` 必须先按主 `SKILL.md` 的路径规则解析，不要拼到 skill 目录下：

```text
python tools/hara/check_stage_json.py --stage stage1 --json output/<run_id>_stage1_derive_mf.json --stage0 output/<run_id>_stage0_function_mapping.json --fix
```
