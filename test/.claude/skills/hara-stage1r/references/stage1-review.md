# Stage 1R: 功能故障评审

目标：单独评审 DeriveMF 的覆盖性和每个功能故障字段的合理性。

## 输入

- Stage 0 JSON：`output/<run_id>_stage0_function_mapping.json`
- Stage 1 JSON：`output/<run_id>_stage1_derive_mf.json`
- Stage 0 中每个功能的 `detail_text`
- Stage 1 使用的知识库

## 输出

- Review JSON：`output/<run_id>_stage1_review.json`
- 修正后的 Stage 1 JSON：仍写回 `output/<run_id>_stage1_derive_mf.json`

## 输出 Schema（严格遵循）

```json
{
  "meta": {
    "run_id": "<run_id>",
    "stage": "stage1_review",
    "target_file": "output/<run_id>_stage1_derive_mf.json",
    "generated_at": "ISO时间戳"
  },
  "overall_result": "pass/failed",
  "issues": [
    {
      "id": "ST1R-001",
      "severity": "error/warning",
      "target": "<row/字段>",
      "description": "<问题描述>",
      "evidence": "<Stage0/Stage1证据>"
    }
  ],
  "fixes": [
    {
      "target_file": "output/<run_id>_stage1_derive_mf.json",
      "target": "<row/字段>",
      "change": "<修正内容>",
      "reason": "<修正原因>"
    }
  ],
  "review_log": [
    {
      "stage": "stage1_review",
      "target": "stage1",
      "result": "pass/failed",
      "issues": [],
      "fixes": [],
      "notes": ""
    }
  ]
}
```

## 检查点

### 基础检查

1. 数量：`derive_mf` 行数必须等于 `function_mapping` 行数。
2. 覆盖：每个 Stage 0 功能必须在 Stage 1 中且只出现一行。
3. 字段完整性：每行必须包含所有故障字段。

### 逐字段评审方法

对每个功能的**每个故障字段**（包括 `nan` 字段），按以下步骤评审：

**第一步：识别功能本质**
- 该功能的核心输出是什么？（力、扭矩、角度、状态、信息、时机）
- 该功能有几个状态/动作方向？

**第二步：重新推理故障适用性（重点）**
- 对于**非 `nan` 字段**：验证描述是否准确、推理是否正确
- 对于**`nan` 字段**：必须按照规则重新推理，确认是否真的不适用
  - 严格按照 `stage1-malfunction.md` 中的判断依据分析
  - 如果功能有相关输出/状态/时机，不能轻易填 `nan`
  - 特别注意：保持/约束/保护类功能的”过小”、二元状态功能的”方向错误”容易被遗漏

**第三步：验证描述质量**
- 描述是否具体说明了故障的异常行为（而不是”功能名 + 引导词”的机械拼接）
- 描述是否限制在当前功能边界内（没有混入相邻功能的故障）
- 描述是否是单一故障效应（没有用”或/以及/同时”打包多个故障）

### `nan` 字段的推理验证格式

对于标记为 `nan` 的字段，按以下格式重新推理：

```
字段：[故障类型]
当前值：nan
重新推理：
- 功能输出/状态/时机：[...]
- 该故障类型是否适用：[根据判断依据分析]
- 如果适用，后果是：[推导人身伤害风险]
- 是否真的不适用：[是/否，理由]
结论：[保持 nan 或修改为具体描述]
```

### 常见错误模式

| 错误类型 | 表现 | 判断方法 |
|---------|------|---------|
| 幅值vs时机混淆 | 把”输出过大”写成”过早”，或把”过晚”写成”过小” | 分析异常是幅值问题还是时机问题 |
| 过小遗漏 | 保持/约束/保护功能的过小填 `nan` | 分析功能输出是否涉及”保持/约束/保护”，这些功能的输出不足都有安全风险 |
| 相邻功能混入 | 当前功能描述了其他功能的故障 | 检查描述中的动作/状态是否属于当前功能边界 |
| 机械拼接 | 只有功能名加引导词，无具体异常行为 | 描述是否说明了”发生了什么异常” |
| 反向错误遗漏 | 二元状态功能方向错误填 `nan` | 检查功能是否有相反的状态/动作方向 |
| 不适用判断错误 | 应该填写的故障填了 `nan` | 分析该故障是否会产生人身伤害风险 |

### 重点检查：过小/过大

对于涉及**保持、约束、保护、制动**的功能：
- 核心输出是力、压力、扭矩、夹紧力时，”过小”几乎总是适用的
- 输出不足会导致无法保持/约束/保护，直接产生安全风险
- 不应轻易填 `nan`，必须有明确理由说明”输出不足不会造成人身伤害”

### 推理过程检查

对**每个故障字段**（无论最终值是具体描述还是 `nan`），验证其推理记录是否完整：

- 推理记录是否存在？（每个字段都必须有）
- `推理.是否有安全风险` 是否存在且为明确的 `是` 或 `否`？
- 功能输出是否正确识别？
- 异常情况是否清晰？
- 后果是否推导到人身伤害风险？

**一致性检查**（重点）：
- 如果 `推理.是否有安全风险` 为 `是`，检查 `derive_mf` 中对应字段是否为 `nan`（应该是具体描述）
- 如果 `推理.是否有安全风险` 为 `否`，检查 `derive_mf` 中对应字段是否为具体描述（应该是 `nan`）
- 发现不一致必须修正，要么修正 `derive_mf` 的值，要么修正推理记录

**重点检查 `nan` 字段**：
- 如果字段值为 `nan`，对应推理记录的 `是否有安全风险` 必须是 `否`
- 推理必须明确说明"为什么不适用"，不能只有简单的"否"

如果发现推理过程不完整或推理与最终值不一致，应要求重新生成。

如果发现问题，修改 Stage 1 JSON，在 review JSON 的 `fixes` 中说明修改内容，并在 Stage 1 JSON `review_log` 中记录通过。
