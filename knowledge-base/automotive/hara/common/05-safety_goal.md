# HARA 安全目标与FTTI生成
如果 ASIL 为 QM，安全目标、安全状态以及FTTI（ms）不需要填写。

## Safety goal 产生
Safety goal 是针对不可接受风险定义的顶层安全目标，应描述必须避免或控制的车辆级危险，而不是具体软件或硬件实现。

- 面向整车级 hazard。
- 覆盖对应 malfunction 的非 QM hazardous events。
- 必须与故障内容一致。故障是“功能丧失/失效”时，安全目标应表达防止该失效导致的车辆级危险；故障是“过小/不足”时，安全目标才表达防止输出不足；故障是“过早/过晚”时，安全目标应表达防止错误时序导致的车辆级危险。
- 必须与危害事件和整车级危害一致，不能把“非预期纵向移动”写成“非预期减速”，也不能把“失效”写成“过大”。
- 表达为“防止、避免、限制、确保”等安全目标语义。
- 不写成底层技术需求，例如具体传感器算法、通信协议或代码逻辑。
- QM 行不生成 safety goal。

推荐表达：
- `防止{车辆级危险行为或危害}。`
- `确保在{关键运行条件}下不发生{车辆级危险行为或危害}。`

## Safe state 产生
Safe state 是系统达到的可观察安全状态。它可以是：
- 保持车辆静止。
- 禁止非预期输出。
- 退化到驾驶员可控状态。
- 释放到不继续扩大危险的状态。
- 抑制错误动作并报警提示驾驶员。

报警、DTC 或 HMI 提示通常只是安全机制的一部分，除非 hazard 本身是信息/告警缺失，否则不应单独作为 safe state。

safe state 应是车辆或功能的外部可观察状态，而不是内部实现动作列表。

safe state 必须能支撑 safety goal，并与故障类型呼应：
- 对拉起失败、夹紧力不足、驻车保持失败：安全状态通常是保持车辆静止、建立或维持足够驻车保持能力，或进入驾驶员可控制的驻车保持状态。
- 对非预期释放：安全状态通常是抑制错误释放、保持驻车夹紧或恢复到安全驻车状态。
- 对非预期拉起/非预期制动：安全状态通常是抑制非预期制动输出，使车辆保持驾驶员可控制的纵向运动状态。
- 对无法释放：安全状态通常是保持车辆静止并提示驾驶员，避免车辆在不受控状态下移动；不要写成“继续夹紧”之外无法解除危险的内部动作。
- 对方向错误：安全状态应抑制错误方向动作，并保持或恢复到与驾驶员请求一致的安全状态。

## FTTI 判断
FTTI（Fault Tolerant Time Interval）是从故障发生到危险无法避免之间允许系统检测、诊断并达到安全状态的时间窗口。

FTTI 判断应考虑：
- 危险演化速度。
- 故障检测时间。
- 系统执行安全反应的时间。
- 驾驶员或道路使用者可用反应时间。
- 当前场景的速度、距离、坡度、交通参与者和空间约束。

FTTI 不应任意给出。理由应说明工程判断依据，例如车辆运动从异常到危险状态的时间、执行器响应时间、诊断周期或场景中的可用反应窗口。

## 判断框架
- 先估计从 malfunction 出现到危险状态不可避免之间的时间窗口。
- 再扣除检测、诊断、仲裁、执行器动作和提示传播所需时间。
- 若依赖驾驶员介入，还应说明驾驶员是否在场、能否感知以及是否有可行动作。

FTTI 应体现“系统必须在多快时间内完成有效控制”，而不是笼统给出经验值。
